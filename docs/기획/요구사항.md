# 요구사항 정의서

## 1. 프로젝트 개요

| 항목 | 내용 |
|------|------|
| 프로젝트명 | 집반찬연구소 MES |
| 작성일 | 2026-06-01 |
| 버전 | v0.1 |

## 2. 배경 및 목적

> 현장 생산 데이터를 실시간으로 수집·관리하고, 생산계획부터 이력추적까지 전 공정을 통합 관리하는 MES(Manufacturing Execution System)를 구축한다.

## 3. 이해관계자

| 역할       | 설명                    |
| -------- | --------------------- |
| 관리자(사무직) | 생산계획 수립, 발주 승인, 마감 처리 |
| 반장       | 간반 관리, 현장 작업 지시       |
| 작업자      | NFC 태그 스캔, 생산 실행 입력   |
| 시스템 관리자  | 디바이스 등록, 기초정보 관리      |

## 4. 작성 규칙

> 아래 질문에 답하며 각 요구사항을 작성합니다.

**기본 정보**
- 이 기능의 이름은 무엇인가?
- 요구사항 ID는? (예: FR-001)
- 우선순위는? (Must Have / Should Have / Nice to Have)
- 이 기능을 사용하는 사람(Actor)은 누구인가?

**기능 정의**
- 누가, 무엇을, 어떻게 할 수 있어야 하는가?
  - "\_\_\_는 \_\_\_을 할 수 있어야 한다" 형태로 한 문장으로 쓸 수 있는가?
- 이 요구사항 하나에 기능이 2개 이상 섞여 있지 않은가?
- 구현 방법이 아닌 결과/목적으로 작성했는가?

**조건**
- 이 기능이 동작하기 위한 사전 조건은 무엇인가?
- 기본 흐름(정상 경로)을 단계별로 나열할 수 있는가?
- 예외 상황은 어떤 것이 있는가? 각각 어떻게 처리되는가?

**수용 기준**
- 이 기능이 "완료"된 것을 어떻게 확인할 수 있는가?
- 수치나 조건으로 표현할 수 있는가? (시간, 횟수, 상태 등)
- 모호한 단어(빠르게, 쉽게, 적절히)를 사용하지 않았는가?

**검토**
- 같은 내용의 요구사항이 이미 존재하지 않는가?
- 관련된 다른 요구사항 ID가 있는가? (참고 링크)
- 기술적으로 구현 가능한 범위인가?

## 5. 기능 요구사항

### 5.1 대시보드

#### FR-001 생산통계 조회

- **설명**: 관리자는 생산성 지표(간반 완료·미완료 현황, 생산 진행률)를 실시간으로 조회할 수 있어야 한다.
- **우선순위**: Must Have
- **Actor**: 관리자
- **사전 조건**: 당일 간반이 1건 이상 생성되어 있어야 한다.
- **기본 흐름**:
  1. 관리자가 대시보드 > 생산통계 화면에 접근한다.
  2. 시스템이 당일 간반 총 수량, 완료 수량, 미완료 수량, 생산 진행률을 표시한다.
- **예외 흐름**:
  - 당일 간반이 없는 경우 → "등록된 생산 데이터가 없습니다" 안내 표시
- **수용 기준**:
  - [ ] 간반 총 수량, 완료 수량, 미완료 수량, 진행률(%)이 화면에 표시된다.
  - [ ] 데이터는 실시간(1분 이내)으로 갱신된다.
- **관련 요구사항**: FR-004 (간반 생성)

---

#### FR-002 현장관제 모니터링

- **설명**: 현장 작업자는 대형 화면을 통해 반별 간반 진행 상태를 실시간으로 확인할 수 있어야 한다.
- **우선순위**: Must Have
- **Actor**: 현장 작업자, 반장
- **사전 조건**: 현장 모니터(5개소) 디바이스가 등록되어 있어야 한다.
- **기본 흐름**:
  1. 현장 모니터가 대시보드 > 현장관제 화면을 표시한다.
  2. 반별 간반 번호, 작업 상태(대기/진행/완료), 담당자, 시작·종료 시간, 진행률이 표시된다.
  3. 작업 상태 변경 시 화면이 자동 갱신된다.
- **예외 흐름**:
  - 디바이스 연결 끊김 → 마지막 데이터 유지 + 연결 오류 표시
- **수용 기준**:
  - [ ] 5개 현장 모니터에서 반별 간반 상태가 실시간으로 표시된다.
  - [ ] 작업 상태 변경 후 30초 이내 화면에 반영된다.
- **관련 요구사항**: FR-004 (간반 생성), FR-038 (디바이스 등록)

---

#### FR-003 사무실관제 모니터링

- **설명**: 관리자는 생산 현황과 온도·습도·CCP 데이터를 2분할 화면으로 실시간 모니터링할 수 있어야 한다.
- **우선순위**: Must Have
- **Actor**: 관리자
- **사전 조건**: HACCP 설비 센서가 연동되어 있어야 한다.
- **기본 흐름**:
  1. 관리자가 대시보드 > 사무실관제 화면에 접근한다.
  2. 화면이 생산 모니터와 품질 모니터 2분할로 표시된다.
  3. 온도(℃), 습도(%) 및 CCP 기준값 대비 현재값이 실시간으로 표시된다.
- **예외 흐름**:
  - 측정값이 기준 범위 초과 → 이상 알림 표시 및 담당자 알림 발송
- **수용 기준**:
  - [ ] 온도, 습도, CCP 데이터가 실시간으로 표시된다.
  - [ ] 이상 발생 시 5분 이내 알림이 전송된다.
- **관련 요구사항**: FR-022 (HACCP 설비)

---

### 5.2 생산계획 PULL

#### FR-004 주문 수신 및 순서 조정

- **설명**: 관리자는 WES에서 생산 주문을 수신하고, 전개 전 생산 순서를 수동으로 조정할 수 있어야 한다.
- **우선순위**: Must Have
- **Actor**: 관리자
- **사전 조건**: WES 시스템과 연동이 완료되어 있어야 한다.
- **기본 흐름**:
  1. 시스템이 WES로부터 주문 리스트를 수동 또는 자동으로 수신한다.
  2. 관리자가 주문 리스트(주문번호, 제품코드, 제품명, 규격, 단위, 수량, 채널, 생산 순서)를 조회한다.
  3. 관리자가 생산 순서를 드래그 또는 입력값으로 조정한다.
- **예외 흐름**:
  - WES 연동 실패 → 오류 메시지 표시, 수동 입력으로 대체 가능
- **수용 기준**:
  - [ ] 쿠팡, 컬리, 자사몰, 다파는 농부, 스마트스토어 등 채널별 주문이 구분 표시된다.
  - [ ] 생산 순서 변경 후 저장 시 즉시 반영된다.
- **관련 요구사항**: FR-005 (전개), FR-006 (간반 생성)

---

#### FR-005 수율 기반 전개 계산

- **설명**: 시스템은 WES 코드를 현장 산 코드로 변환하고, 전처리·냉각·포장 3종 수율을 적용하여 소요량을 자동 계산해야 한다.
- **우선순위**: Must Have
- **Actor**: 관리자
- **사전 조건**: 품목별 수율 기준값이 등록되어 있어야 한다. (FR-035)
- **기본 흐름**:
  1. 관리자가 전개 단계를 실행한다.
  2. 시스템이 WES 코드 → 산 코드로 변환한다.
  3. 시스템이 전처리·냉각·포장 수율(%)을 적용하여 환산 수량을 계산한다.
  4. 관리자가 전개 결과를 확인하고 필요 시 보정한다.
- **예외 흐름**:
  - 수율 기준값 미등록 품목 → 해당 품목 표시 및 수동 입력 요청
- **수용 기준**:
  - [ ] 전처리, 냉각, 포장 3종 수율이 모두 반영된 환산 수량이 계산된다.
  - [ ] 수동 보정값 입력 후 저장이 가능하다.
- **관련 요구사항**: FR-004 (주문), FR-035 (수율 기초정보)

---

#### FR-006 간반 자동 생성 및 출력

- **설명**: 시스템은 반별 간반을 자동 생성하고, 간반 번호를 반별 개별 순번으로 부여하며, 관리자는 간반을 출력할 수 있어야 한다.
- **우선순위**: Must Have
- **Actor**: 관리자
- **사전 조건**: 전개 계산이 완료되어 있어야 한다. (FR-005)
- **기본 흐름**:
  1. 시스템이 반별 간반을 자동 생성한다.
  2. 간반번호, 반 구분, 제품명, 수량, 단위, 담당 공정이 자동 입력된다.
  3. 관리자가 간반을 출력한다.
- **예외 흐름**:
  - 프린터 연결 오류 → 오류 메시지 표시, 재시도 버튼 제공
- **수용 기준**:
  - [ ] 반별 간반 번호가 개별 순번으로 중복 없이 부여된다.
  - [ ] 출력 여부가 시스템에 기록된다.
- **관련 요구사항**: FR-005 (전개), FR-007 (생산 실행)

---

#### FR-007 생산 실행 및 NFC 스캔 등록

- **설명**: 작업자는 NFC 태그 스캔으로 간반별 작업 시작·종료를 등록할 수 있어야 한다.
- **우선순위**: Must Have
- **Actor**: 작업자, 반장
- **사전 조건**: NFC 태그와 현장 디바이스가 등록되어 있어야 한다. (FR-037, FR-038)
- **기본 흐름**:
  1. 반장이 간반별 담당자를 지정한다.
  2. 작업자가 NFC 태그를 스캔하여 작업을 시작한다.
  3. 시스템이 시작 시간을 기록한다.
  4. 작업자가 완료 후 NFC 태그를 재스캔하여 종료를 등록한다.
  5. 시스템이 종료 시간과 작업 상태를 기록한다.
- **예외 흐름**:
  - NFC 스캔 실패 → 수동 입력으로 대체 가능
- **수용 기준**:
  - [ ] NFC 스캔 1회로 작업 시작·종료 등록이 가능하다.
  - [ ] 간반번호, 담당자, 시작·종료 시간, 작업 상태가 저장된다.
- **관련 요구사항**: FR-006 (간반 생성), FR-037 (NFC 태그)

---

#### FR-008 일일 생산 마감

- **설명**: 관리자는 일일 생산 마감을 처리하고, 사진·청소 상태·에너지 검침값을 기록할 수 있어야 한다.
- **우선순위**: Must Have
- **Actor**: 관리자
- **사전 조건**: 당일 모든 간반 작업이 완료 상태여야 한다.
- **기본 흐름**:
  1. 관리자가 생산계획 PULL > 5단계: 완료 화면에 접근한다.
  2. 관리자가 사진을 첨부하고 청소 상태를 입력한다.
  3. 관리자가 가스·전기·수도 검침값을 입력한다.
  4. 관리자가 마감을 확정한다.
  5. 시스템이 마감 확정 이후 수정을 불가 처리한다.
- **예외 흐름**:
  - 미완료 간반 존재 시 → 마감 불가 안내 및 미완료 목록 표시
- **수용 기준**:
  - [ ] 마감 확정 후 해당 일자 데이터 수정이 불가능하다.
  - [ ] 사진, 청소 상태, 검침값 3종이 모두 저장된다.
- **관련 요구사항**: FR-007 (생산 실행)

---

#### FR-009 추가생산 처리

- **설명**: 관리자는 간반 출력 전·후 상황에 따라 추가 생산을 등록하거나 간반을 교체 처리할 수 있어야 한다.
- **우선순위**: Must Have
- **Actor**: 관리자
- **사전 조건**: 해당 일자의 간반이 생성되어 있어야 한다.
- **기본 흐름**:
  1. 관리자가 추가생산 화면에 접근한다.
  2. 간반 출력 전인 경우: 기존 데이터를 수정한다.
  3. 간반 출력 후인 경우: 간반 교체 처리를 진행한다.
  4. 추가 제품코드, 수량, 처리 방식, 사유, 요청 시간을 입력한다.
- **예외 흐름**:
  - 이미 완료된 간반에 추가생산 요청 → 불가 안내 표시
- **수용 기준**:
  - [ ] 출력 전/후 처리 방식이 구분된다.
  - [ ] 처리 사유 및 요청 시간이 기록된다.
- **관련 요구사항**: FR-006 (간반 생성)

---

#### FR-010 추가불출 요청

- **설명**: 작업자는 작업 중 재료 부족 시 추가 수량을 요청할 수 있어야 한다.
- **우선순위**: Must Have
- **Actor**: 작업자, 반장
- **사전 조건**: 해당 간반이 진행 중 상태여야 한다.
- **기본 흐름**:
  1. 작업자가 추가불출 화면에 접근한다.
  2. 간반번호, 자재코드, 기준 수량, 추가 수량, 요청자를 입력한다.
  3. 시스템이 해당 간반에 추가 수량을 매칭 입력한다.
- **예외 흐름**:
  - 완료된 간반에 대한 추가불출 요청 → 불가 안내 표시
- **수용 기준**:
  - [ ] 요청 시간과 요청자가 기록된다.
  - [ ] 해당 간반에 추가 수량이 매칭된다.
- **관련 요구사항**: FR-007 (생산 실행)

### 5.3 생산계획 PUSH

#### FR-011 주간 생산계획 수립

- **설명**: 관리자는 주간 단위로 생산 계획을 수립하고 확정할 수 있어야 한다.
- **우선순위**: Should Have
- **Actor**: 관리자
- **사전 조건**: 품목 DB와 BOM이 등록되어 있어야 한다. (FR-028, FR-030)
- **기본 흐름**:
  1. 관리자가 주간 생산계획 화면에서 계획 주차를 선택한다.
  2. 제품코드, 제품명, 계획 수량을 입력한다.
  3. 시스템이 MRP 기반 소요량을 자동 계산한다.
  4. 관리자가 계획을 확정한다.
  5. 확정된 계획이 생산 실행으로 이동된다.
- **예외 흐름**:
  - BOM 미등록 품목 → 해당 품목 표시 및 소요량 수동 입력 요청
- **수용 기준**:
  - [ ] 계획 주차, 제품코드, 계획 수량, 확정 여부, 작성자가 저장된다.
  - [ ] 확정 시 생산 실행 화면에서 조회 가능하다.

  - [ ] 연결 상태가 화면에 표시된다.
- **관련 요구사항**: FR-037 (NFC 태그)

---

### 5.10 시스템 관리

#### FR-037 NFC 태그 등록 및 관리

- **설명**: 시스템 관리자는 NFC 태그를 등록하고 설비·위치·품목과 연결 설정할 수 있어야 한다.
- **우선순위**: Must Have
- **Actor**: 시스템 관리자
- **사전 조건**: 없음
- **기본 흐름**:
  1. 시스템 관리자가 NFC 태그 ID를 등록한다.
  2. 연결 대상(설비/위치/품목)과 연결 작업을 설정한다.
  3. 등록일과 상태를 저장한다.
- **예외 흐름**:
  - 태그 ID 중복 등록 → 오류 메시지 표시
- **수용 기준**:
  - [ ] 태그 ID와 연결 대상이 매핑된다.
  - [ ] 도미노 스캔 대체 기능으로 동작한다.
- **관련 요구사항**: FR-007 (생산 실행), FR-036 (도미노 TTO)

---

#### FR-038 현장 디바이스 등록

- **설명**: 시스템 관리자는 현장 디바이스 3종을 등록하고 작업자와 매핑할 수 있어야 한다.
- **우선순위**: Must Have
- **Actor**: 시스템 관리자
- **사전 조건**: 작업자 정보가 등록되어 있어야 한다. (FR-032)
- **기본 흐름**:
  1. 시스템 관리자가 디바이스 유형(공용 간반 출력용/반장 개별 지급용/사무직용)을 선택한다.
  2. 디바이스 ID, 위치, 배정 작업자를 입력한다.
  3. 등록일과 상태를 저장한다.
- **예외 흐름**:
  - 이미 배정된 디바이스에 다른 작업자 매핑 → 기존 배정 해제 확인 팝업 표시
- **수용 기준**:
  - [ ] 디바이스 3종이 유형별로 구분 등록된다.
  - [ ] 디바이스-작업자 매핑이 저장된다.
- **관련 요구사항**: FR-007 (생산 실행), FR-002 (현장관제)
